## Development setup

The project supports two equivalent setups â€” pick whichever you prefer.

### With [uv](https://docs.astral.sh/uv/) (recommended â€” fastest, reproducible)

```bash
git clone git@github.com:karthicraghupathi/django_rapyd_modernauth.git
cd django_rapyd_modernauth
uv sync --all-extras
echo "DJANGO_SECRET_KEY=local-dev-secret" > .env
```

`uv sync --all-extras` reads `pyproject.toml` and `uv.lock`, creates a `.venv`, and installs every pinned dependency in a single fast pass. Activate with `source .venv/bin/activate` if you want commands like `tox` on `PATH`, or prefix with `uv run` (e.g. `uv run tox p`).

### With pip + venv (works without uv installed)

```bash
git clone git@github.com:karthicraghupathi/django_rapyd_modernauth.git
cd django_rapyd_modernauth
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
echo "DJANGO_SECRET_KEY=local-dev-secret" > .env
```

For an exactly-pinned dev environment matching CI, swap the editable install for `pip install -r requirements-dev.txt && pip install -e . --no-deps`.

### Then enable the pre-commit hooks

```bash
pre-commit install
```

The `[dev]` extra (defined under `[project.optional-dependencies]` in `pyproject.toml`) installs `python-dotenv`, `tox`, `bump-my-version`, `build`, `twine`, and `pre-commit`. The `.env` file is required by the bundled test project's settings module (it loads `DJANGO_SECRET_KEY` via `python-dotenv`) and is gitignored.

> **Note**: `requirements-dev.txt` is **autogenerated by `uv export`** via a pre-commit hook. Never hand-edit it. To add a dev dependency, edit `[project.optional-dependencies] dev` in `pyproject.toml` and let the hook (or `uv lock && uv export --frozen --all-extras --no-emit-project --output-file=requirements-dev.txt`) regenerate the lock and requirements files.

## Linting and formatting

The pre-commit hooks run automatically on `git commit`. To run them all once against the whole tree:

```bash
pre-commit run --all-files
```

This runs `ruff check --fix`, `ruff format`, and the standard `pre-commit-hooks` checks (trailing whitespace, end-of-file fixer, YAML well-formedness, etc.). Configuration for ruff lives in `pyproject.toml` under `[tool.ruff]`.

## Cutting a release

Releases are fully automated. The local steps are:

```bash
# 1. Confirm tree is clean and tests are green
git status
tox p

# 2. Bump the version (creates the version commit + git tag automatically)
bump-my-version bump patch       # 0.0.6 â†’ 0.0.7  (bug-fix release)
bump-my-version bump minor       # 0.0.6 â†’ 0.1.0  (Django LTS support changes, behavior changes, etc.)
bump-my-version bump major       # 1.0.0 â†’ 2.0.0  (breaking)

# 3. Push commit + tag
git push origin main --follow-tags
```

That's it on the local side. The rest happens on GitHub:

1. The `tests` workflow runs the matrix on the bump commit.
2. The `release` workflow fires on the new `v*` tag and creates a GitHub Release with auto-generated notes (PR titles since the previous tag).
3. The `publish` workflow fires on the release publication, builds the sdist and wheel via `uv build`, and uploads to PyPI through OIDC trusted publishing.

No API tokens, no manual `twine upload`. Watch the Actions tab to see the cascade.

A `bump minor` is also the right call when the package's *behavior* changes â€” even if the public API surface is unchanged â€” because existing consumers need to be aware of the change before upgrading. "Minor" is not just for new features.

`bump-my-version` configuration lives under `[tool.bumpversion]` in `pyproject.toml`. Use `bump-my-version bump <part> --dry-run --verbose` to preview the exact files and lines that would change.

## Reporting bugs

Use the [issue tracker](https://github.com/karthicraghupathi/django_rapyd_modernauth/issues) for non-security bug reports and feature requests. A useful report includes:

- The version of `django-rapyd-modernauth` and Django you're using
- The Python version
- A minimal reproduction (or pointer to a failing test)
- What you expected vs. what happened
